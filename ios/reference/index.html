Summary

The NativeXO ( Native Checkout ) SDK is a set of libraries and classes to bring a native experience to a checkout on mobile phone. This is done in a two-fold manner:

  • 1st Party Experience: Modules that will be part of Venice, and other first party mobile apps
  • 3rd Party Experience: Libraries and assets that a merchant may include as part of their app

Benefits

  1. Allows Venice and other first party apps to support web-based merchant checkouts
  2. Allows native merchant apps to use PayPal
  3. Allows a safe login on merchant apps via Venice
  4. Faster Checkout ðŸ™‚

Supported Platforms

This is the incarnation that supports Swift and Objective C-based platforms.

Currently, the SDK only supports iOS 11.0 and above. Some features may be limited to some versions of the system. The support is defined within:

  • Package.swift
  • .swiftlint.yml

Support for tvOS, watchOS, and macOS are theoretically possible but have not been implemented.

Prerequisites

Cocoapods:

We currently use Cocoapods to utilize dependencies for PayPalCheckout, to install Cocoapods ensure that you have ruby installed which will come prepackaged on older Macbook machines, and in new units you might have to install ruby yourself.

To check if you have the ruby gem system installed simply run:

which gem

If you get valid output, then you are good to go. Otherwise you can utilize Homebrew to install ruby:

brew install ruby

Next install cocoapods if you do not already have it installed:

sudo gem install cocoapods

SwiftLint:

Ensure that you have SwiftLint installed as we utilize it within our project.

To install run:

brew install swiftlint

Our Xcode project already has a Run Phase which integrates in SwiftLint so the only prerequisite is installing via Homebrew.

  • Provider - Checkout provider, e.g. PayPal, Venmo, etc.
  • Paysheet - UI specific to the checkout process

Structure

The project is broken down as follows:

  1. The Core layer contains shared contracts and functionality protocols
  2. The IO layer contains network connectivity, and can generate objects from GraphQL
  3. The Events layer handles notifications and updates between SDK classes, such as updating viewmodels
  4. The Platform layer communicates with the PayPal platform front end and checkout lifecycle
  5. The UI layer implements the UI and graphics used in the paysheet
  6. The Checkout layer is the only layer presented to merchant apps, revealing only what the merchant needs

Core Layer

The core layer contains the PayPalCheckoutCore namespace and is reserved for protocol definitions and canonical types. This namespace should be implementation agnostic, and only contain functionality that is extensible and broadly applicable.

Example: The core defines the canonical ShippingAddress contract. It should never determine where it comes from; merely its structure.

Note: The module in this layer should never depend on any of the other layers in this project.

IO Layer

The IO layer contains the PayPalCheckoutIO namespace and is reserved for general purpose communications, serializing, and deserializing.

Events Layer

The Events layer deals with interobject communication.

Platform Layer

The Platform layer builds upon the Core, IO, Events layers to manage the application’s state and relationship with the PayPal front tier services.

UI Layer

The UI layer implements the UIViews and view models necessary for paysheets and their management

Checkout Layer

The Checkout layer encapsulates all the other layers, not only tying the pay sheet UI to PayPal’s services, but also providing a unified interface for the merchant application.
